# Сетевой уровень MusicBand (UDP, чанки, клиент и сервер)

Документ описывает, как приложение передаёт запросы и ответы между клиентом и сервером: от объекта в памяти до байтов в UDP‑сокете и обратно.

---

## 1. Роль UDP и чем оно отличается от TCP

**UDP (User Datagram Protocol)** — протокол без установления соединения и без гарантий доставки и порядка пакетов. Каждая единица передачи — **датаграмма**: независимый пакет с адресом отправителя и получателя.

В этом проекте:

- Один **логический запрос** клиента = один объект `AbstractRequest`, сериализованный в байты.
- Один **логический ответ** сервера = один объект `AbstractResponse`, сериализованный в байты.
- Если сериализованное сообщение **длиннее одного UDP‑пакета**, оно **режется на фрагменты** (чанки), отправляется несколькими датаграммами и **собирается** на другой стороне.

Почему не TCP: по условию лабораторной используется именно UDP; ограничение размера одной датаграммы (~65 КБ полезной нагрузки у IPv4) обходится фрагментацией на уровне приложения.

## 2. Сериализация: `BinaryProtocol` и Java Serialization

Класс `common.util.BinaryProtocol`:

1. **`serialize(Serializable)`** — пишет объект в `ObjectOutputStream` → `ByteArrayOutputStream` → получает **массив байт** всего графа объектов (запрос или ответ).
2. **`deserialize(byte[], Class<T>)`** — читает из `ObjectInputStream` и проверяет, что тип совпадает с ожидаемым.

Используется **стандартная Java Serialization** (`Serializable`, `serialVersionUID` у классов протокола). Это значит:

- Клиент и сервер должны иметь **одинаковые классы** запросов/ответов на classpath (модуль `common`).
- Версии классов должны быть совместимы (поля не ломать произвольно).

Сетевой слой оперирует **только байтами**; смысл байтов задаёт `BinaryProtocol`.

## 3. Фрагментация: `SimpleUdpStreamFraming` (чанки)

Файл `common.net.udp.SimpleUdpStreamFraming`.

### 3.1. Константы

| Константа | Значение | Смысл |
|-----------|----------|--------|
| `PACKET_SIZE` | 1024 | Максимальный размер **одной** датаграммы полезной схемы (данные + маркер). |
| `DATA_SIZE` | 1023 | Максимум байт **полезной нагрузки** в одном пакете (без маркера). |
| `MAX_MESSAGE_BYTES` | 50 MB | Защита от слишком большого сообщения при сборке. |

### 3.2. Формат одной датаграммы

Структура пакета:

```
[ байты 0 .. (n-1) ] [ маркер 1 байт ]
```

- `n` от 0 до 1023 — кусок сериализованного сообщения.
- **Последний байт** — маркер:
  - `0` (`CONTINUATION`) — после этого пакета идут ещё фрагменты **того же** сообщения;
  - `1` (`END`) — это **последний** фрагмент; после сборки всех предыдущих кусков + этого (без маркера) получается **полный** массив байт одного `serialize(...)`.

Пустое сообщение (на практике не встречается для запросов) обрабатывается как один пакет только с маркером `END`.

### 3.3. Отправка: `split(byte[] message)`

Алгоритм:

1. Пока не обработан весь `message`, взять до 1023 байт подряд.
2. Сформировать массив длины `n+1`: скопировать кусок, в конец записать `0` или `1` (если это последний кусок — `1`).
3. Вернуть `List<byte[]>` — по одному UDP‑пакету на элемент списка.

Порядок пакетов на проводе должен совпадать с порядком в списке (клиент и сервер отправляют последовательно в цикле).

### 3.4. Приём: `StreamReceiver.feed(datagram, length)`

1. Взять последний байт датаграммы — маркер; всё, что перед ним, — данные фрагмента.
2. Данные **дописать** во внутренний `ByteArrayOutputStream`.
3. Если маркер `END` — вернуть **полный** массив байт сообщения, очистить буфер.
4. Если маркер `CONTINUATION` — вернуть `null` (ждём следующие датаграммы).

Ограничения модели:

- Нет номера «сессии» в пакете: предполагается, что **одновременно** собирается только одно сообщение от одного адреса (см. сервер ниже).
- Потеря или перемешивание датаграмм в реальной сети протокол **не исправляет** (для локальной сети/localhost обычно достаточно).

## 4. Сервер: `UdpServer`, `ChunkedUdpSocketTransport`, `IncomingDatagramProcessor`

### 4.1. `UdpServer`

- Открывает **`DatagramSocket`** на порту.
- Создаёт **`ChunkedUdpSocketTransport`** и **`IncomingDatagramProcessor`** (реестр команд + диспетчеризация).
- В бесконечном цикле:
  1. `transport.receiveComplete()` — блокируется, пока не соберётся **целое** сообщение (все чанки).
  2. `incomingProcessor.process(payload, replyAddress)` — перед dispatch в контекст записывается адрес клиента; из байт делается `AbstractRequest`, вызываются обработчики, получается `AbstractResponse`.
  3. `transport.sendComplete(replyAddress, response)` — сериализует ответ и снова режет на чанки, шлёт на адрес клиента.

Ошибки `ProtocolException` при приёме/разборе чанков логируются, ответ клиенту в этом цикле не формируется (клиент по таймауту поймёт сбой).

### 4.2. `ChunkedUdpSocketTransport`

- **`receiveComplete()`**:
  - В цикле `socket.receive(DatagramPacket)`.
  - Копирует **фактическую** длину `packet.getLength()` (важно: не весь буфер массива).
  - Запоминает **`expectedSender`**: первый фрагмент задаёт адрес клиента; все следующие фрагменты **должны** быть с того же `SocketAddress`, иначе сброс сборки и `ProtocolException` (защита от смешивания двух клиентов в одной «склейке»).
  - Передаёт байты в `StreamReceiver.feed`.
  - Когда вернулся полный массив — сбрасывает `expectedSender`, возвращает `Received(payload, replyAddress)`.

- **`sendComplete(to, Serializable)`**:
  - `BinaryProtocol.serialize(object)` → `SimpleUdpStreamFraming.split(bytes)` → для каждого куска `socket.send`.

Имя класса исторически «Chunked»; по сути это **потоковая** нарезка с маркером конца, без отдельных ID фрагментов.

### 4.3. `IncomingDatagramProcessor`

- В `process(payload, clientAddress)` в `CollectionCommandContext` выставляется **`currentClient`**, после обработки сбрасывается (нужно для стека скриптов по адресу UDP).
- `BinaryProtocol.deserialize(payload, AbstractRequest.class)`.
- При ошибке десериализации — `ErrorResponse` с текстом (ид запроса может быть 0, если заголовок не прочитан).
- При успехе — `ServerCommandRegistry.dispatch(request)` — паттерн Command на сервере.

## 5. Клиент: `UdpClient`

### 5.1. Компоненты

- **`DatagramChannel`** в **неблокирующем** режиме (`configureBlocking(false)`).
- **`Selector`** с интересом `OP_READ` на канале.
- Адрес сервера: `InetSocketAddress(host, port)`.

### 5.2. Отправка `sendAndReceive(request)`

1. **Сериализация** запроса: `BinaryProtocol.serialize(request)` → `byte[] payload`.
2. **Фрагментация**: `SimpleUdpStreamFraming.split(payload)` → список UDP‑пакетов.
3. **Отправка**: для каждого куска `ByteBuffer.wrap(chunk)` и цикл `channel.send(buffer, serverAddress)` пока буфер не опустошён (на случай частичной отправки API).

4. **Приём ответа**:
   - Создаётся новый `StreamReceiver` (чистое состояние для **нового** сообщения).
   - Буфер приёма `ByteBuffer.allocate(PACKET_SIZE)` (1024 байта).
   - До истечения **`timeoutMs`**:
     - `selector.select(wait)` с оставшимся временем до дедлайна.
     - При `readable`: `channel.receive(in)`; принимаются только датаграммы с **`from.equals(serverAddress)`** (чужой трафик игнорируется).
     - `receiver.feed(data, length)`; если вернулся полный массив — **`BinaryProtocol.deserialize(..., AbstractResponse.class)`** и возврат из метода.
   - Если время вышло — `TimeoutException`.

### 5.3. Почему Selector, а не блокирующий `receive`

Чтобы в одном потоке **ждать ответ с таймаутом**, не блокируя навсегда: `select` возвращает управление по таймеру, можно показать пользователю ошибку «сервер не ответил».

## 6. Сквозной путь одной команды консоли

1. Пользователь вводит строку → `ClientLineDispatcher` → конкретная `ClientCommand` строит `AbstractRequest` (например `ShowRequest`).
2. `ClientCommandContext.send(...)` или `sendAndPrint` → `UdpClient.sendAndReceive(request)`.
3. Запрос сериализуется, режется, уходит UDP‑датаграммами на сервер.
4. Сервер собирает байты → десериализует запрос → `dispatch` → обработчик → `AbstractResponse`.
5. Ответ сериализуется, режется, уходит обратно на `replyAddress` из первой датаграммы (у UDP адрес клиента известен из каждого пакета).
6. Клиент собирает ответ, десериализует, печатает `getMessage()` и т.д.

Команда **`execute_script`**: файл читается **на клиенте** (`ExecuteScriptRunner` + `Scanner` по пути на диске клиента). Перед циклом по строкам клиент шлёт **`ScriptSessionBeginRequest`** (токен = нормализованный абсолютный путь к файлу), после — **`ScriptSessionEndRequest`** с тем же токеном. Сервер только ведёт **стек токенов по адресу клиента** и отклоняет повторный вход в тот же файл (рекурсия). Каждая строка скрипта, которая является командой (`show`, `insert 1`, …), обрабатывается как обычная консольная команда: отдельный запрос и отдельный ответ по UDP (через те же чанки).

Пока выполняется скрипт, **`InputHandler`** клиента временно переключается на тот же `Scanner`, чтобы многострочные команды (`insert`, `update`, …) читали поля из **файла скрипта**, а не с клавиатуры.

## 8. Скрипт и сеть

Сетевой слой по-прежнему передаёт только **сериализованные запросы/ответы**. Формат строк в файле скрипта и порядок полей для `insert` / `update` / … описаны в **`docs/SERVER_SCRIPT.md`** (файл лежит на **клиенте**; сервер файл не открывает).

---

## 9. Ограничения и типичные сбои

| Проблема | Поведение |
|----------|-----------|
| Потеря UDP‑пакета | Сборка не завершится; клиент получит **таймаут**. |
| Два клиента шлют чанки одновременно на сервер | `ChunkedUdpSocketTransport` сбросит сборку при смене адреса → `ProtocolException` на сервере. |
| Сообщение > 50 MB | `ProtocolException` при split или при сборке. |
| Несовпадение версий `common` | `DeserializationException` / ошибка класса. |

Этого достаточно, чтобы объяснить на защите весь путь от команды в консоли до UDP и обратно.
